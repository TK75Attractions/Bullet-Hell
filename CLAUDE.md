# Bullet-Hell プロジェクト指示書

Unity `6000.3.9f1` の弾幕ゲーム。現在の主対象は石工ステージで、音ハメ、タイル落下/着地、コンベア移動、動画レビューの流れが重要。

## セッション開始時に読むもの

- `Docs/claude-code-handoff.md` — 全体の引き継ぎ。作成日付が古い場合、リポジトリの現状と差分がある前提で読む
- `Docs/stone-stage-handoff.md` — 石工ステージの詳細
- BulletBuffer JSON を編集する前に、必ず `Docs/BulletBufferContext.md`

## モデル別の実装方針

Fable メイン運用時のサブエージェント切り出し方針、Opus メイン時の挙動は D: ドライブ共通ルール（`D:\CLAUDE.md`）に移動した。そちらを参照。

## 不変のルール

### 座標・データ仕様

- プレイエリアは `32 x 18`、左下 `(0,0)`、右上 `(32,18)`
- ステージ側 spawner の角度は度、BulletBuffer JSON の極座標角はラジアン
- spawner の `clipName` は JSON トップレベルの `name` と一致させる

### Unity MCP

- 変更系の呼び出しと Play Mode への突入を並列に行わない（UnityMCP が不安定になる）
- スクリプト編集後は、コンパイル完了とエラー有無を確認してから次の作業に進む
- 石工ステージの動作確認は、先に Play Mode に入ってから `Tools/Bullet Hell/Debug/Start Stone Stage` を実行する
- 状態確認には `Tools/Bullet Hell/Debug/Dump Stone Debug State` が使える

### Git

- 作業開始時に必ず `git status -sb`。ワークツリーには意図的な未コミット変更が混在していることが多い
- dirty なファイルを勝手に revert しない（過去セッションの仕掛かり作業の可能性がある）
- staging は狭く、対象ファイルだけを add する。`git add .` は使わない
- `Assets/Screenshots/`、`Assets/_Recovery/`、`Recordings/`、`.oracle-output/` はコミットしない
- 大きな変更の前には、基準点としてローカル commit を作ってよい
- push・公開・force push はユーザー確認を取ってから

### 検証

- 実装後は説明で終えず、コンパイル → Play Mode → デバッグメニューで実挙動を確認する
- 音ハメ・演出系の変更は、Unity Recorder で録画し Oracle/Gemini の動画レビューにかける流れがある
- 検証できなかった項目は、未検証であることを明記する
- **録画やビルドなど長時間処理の完了待ちで background job / watcher / 完了通知を使わない**。`-p`（非対話）実行では通知が届かず、待ったままセッションが終了して成果が宙に浮く事故が繰り返し起きている。必ず同期実行（自前のポーリングループで sleep しながらファイルサイズ安定等を確認）で完了まで見届けてから次に進む

## 報告スタイル

- 日本語で報告する。技術名・ファイル名・コマンドは英語のまま
- 最後に、変更点・検証結果・未解決点を短くまとめる


## レビュー恒久ルール（ユーザーが繰り返し指摘している事項。毎回の実装前に必ず守る）

ユーザーのレビュー（`Instructions/REVIEW-NOTES.md`）で何度も同じ指摘が出ている。以下は**個別レビューに書かれていなくても常に適用する**。違反したまま録画を出すとユーザーのレビュー負担が増える。

### 位置・整合（最頻出）

- **予告と実体の位置は必ず一致させる**（warn ↔ spawn/drop/settle を機械照合し d=0 を確認）。実体が動く演出なら予告も対応して動かす
- **めり込み・浮きの禁止**: ブロックは地面や他ブロックに正確に接地させる。上端・右端など画面端の隙間や余りも作らない（右端の列まで使い切る）
- **落下は真っ直ぐ**: 斜め落下・着地間際の水平ドリフトは禁止。最初から重ならない x に配置して真下に落とす

### 「タイル状ランダム配置」の定義（2回再指摘あり）

- 画面幅を均等なタイル（セル）に区切り、**各タイルに1個ずつ**配置し、**タイル内の位置だけランダム**にする
- 結果として「全体は均一に埋まるが、整列して見えない」状態が正解。行・列・等間隔・対称などの規則性が見えたら NG
- 実装後は必ずフレームを抽出し、規則性が残っていないか自分の目で確認する

### 速度・タイミング

- **連動して動くもの同士の速度は厳密に一致させる**（例: コンベア本体とその模様。数値を同一ソースから導出する）。この注意は複数回出ている
- 予告は本体より十分先行して出し、**本体が出現したら同時に消す**。ただし早すぎる予告も NG（目安はユーザーのレビューで調整）
- 落下着地後は少し静止時間を置いてから次の動きへ

### 表示の後始末

- **親要素が消えたら付随要素（模様・マーカー・エフェクト）も同時に消す**。「模様だけ残ってる」は既出の指摘
- 破片・エフェクトは途中で不自然に消さない（画面外に出るか着地するまで生存させる）
- ループする表示はシームレスにループさせる（重なり・途切れを作らない）

### 録画・検証

- 録画は毎回「作ってある部分全体」を音声付きで `Recordings/` に残す
- 修正した箇所は録画のフレーム抽出で実挙動を自己検証してから完了報告する

## 短期的な指示

個別の修正指示は `Instructions/REVIEW-NOTES.md` の未処理 `- [ ]` 項目が唯一のキューになっている（このセクションに書かれていた旧指示は全て対応済み）。処理したら `[x]` 化する。
